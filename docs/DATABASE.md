# AGR Database Structure

Разработанная структура базы данных на основе анализа старой системы.

## Основные таблицы

### 1. **users** - Таблица пользователей
```sql
- id (Primary Key)
- name: Имя пользователя
- login: Логин (уникальный)
- pass: Пароль (MD5 хеш)
- role: Роль пользователя
- created_at, updated_at: Временные метки
```

### 2. **objects** - Основная таблица объектов (дома, здания)
```sql
Идентификаторы:
- id (Primary Key)
- IMEI: Идентификатор GSM модема (уникальный)
- IMEI2: Второй GSM модем (опционально)

Местоположение и информация:
- address: Адрес объекта
- City: Город
- lat, lon: GPS координаты
- Contact: Контактная информация

GSM и коммуникация:
- GSMNR: GSM номер
- GSMNR2: Второй GSM номер
- GSMSERIAL: Серийный номер GSM модема
- GSMSERIAL2: Серийный номер второго модема
- pin1, pin2, puk1, puk2: PIN и PUK коды
- KeyCode: Ключевой код
- lastSession: Время последней сессии

Конфигурация устройства:
- dtype: Тип устройства (1, 2, 3 и т.д.)
- Devqtty: Количество устройств
- RadioDevQty: Количество радио устройств
- MainRadio: Основное радио устройство
- ver: Версия прошивки
- status: Статус (1 = активно)

Тариф и платежи:
- Company: ID компании/организации
- manager: ID менеджера
- packet: Тариф/пакет услуги
- traffic: Трафик данных
- callCnt: Количество звонков
- summ: Сумма счета
- fee: Комиссия
- AddFee: Дополнительная комиссия
- kuluM2: Расход на м2

Отопление и показания:
- Description, Description2: Описание объекта
- selDate: Дата проверки/выбора
- saveHval: Сохранять значения отопления
- m2_andur: Площадь по учету (м2)
- dataToPage: Отправлять данные на страницу
- AlgLopp: Алгоритм расчета
```

### 3. **objects_install_data** - Данные установки объектов
```sql
- id (Primary Key)
- oid: ID объекта (Foreign Key -> objects.id)
- devid: ID устройства
- location: Местоположение устройства
- devtype: Тип устройства
- dnType: Тип трубопровода
- len: Длина
- foto1, foto2: Фото
- place1, place2: Место установки
- comment: Комментарий
```

### 4. **log** - Общий логирование системы
```sql
- id (Primary Key)
- Content: Содержание лога
- created_at, updated_at: Временные метки
```

### 5. **csq** - Таблица качества сигнала GSM
```sql
- id (Primary Key)
- object: ID объекта (Foreign Key -> objects.id)
- csq: Значение качества сигнала (0-31)
- created_at, updated_at: Временные метки
```

## Динамические таблицы (создаются автоматически для каждого объекта)

При добавлении нового объекта система автоматически создает следующие таблицы:

### **object_{object_id}** - Список устройств объекта
```sql
- id: Уникальный идентификатор записи
- devid: ID устройства (уникальный в рамках объекта)
- location: Местоположение
- devtype: Тип устройства
```

### **lastdata_{object_id}** - Последние показания приборов
```sql
- devid: ID устройства (Primary Key совместно)
- date: Дата и время последнего показания
- value: Последнее значение (для воды/газа)
- mvalue: Измененное значение
- prevVal: Предыдущее значение
- tariff_1, tariff_2: Показания по тарифам (для электросчетчиков)
- tariff_1_mvalue, tariff_2_mvalue: Измененные значения по тарифам
- tariff_1_prevVal, tariff_2_prevVal: Предыдущие значения по тарифам
- inserterd: Дата вставки
- error: Сообщение об ошибке
- errorDate: Дата ошибки
- statDate: Дата статуса
```

### **mlog_{object_id}** - История показаний (логирование)
```sql
- id: Уникальный идентификатор
- devid: ID устройства
- date: Дата показания
- value: Значение показания
```

### **flat_{object_id}** - Список квартир/секций объекта
```sql
- id: ID квартиры
- location: Номер квартиры или местоположение
- size: Площадь (м2)
```

### **heat_{object_id}** - Данные отопительных приборов
```sql
- id: Уникальный идентификатор
- devid: ID прибора отопления
- power: Мощность
- cof: Коэффициент
- size: Размер
- description: Описание
```

## Модели Eloquent

Созданы следующие Eloquent модели для работы с БД:

- `App\Models\User` - Пользователи
- `App\Models\Object` - Объекты
- `App\Models\ObjectInstallData` - Данные установки
- `App\Models\Log` - Логирование
- `App\Models\Csq` - Качество сигнала

## Отношения между моделями

```
User (1) -----> (M) Objects (как manager)
          
Object (1) -----> (M) ObjectInstallData
        |
        +-----> (M) Csq
```

## Индексы для оптимизации

Созданы индексы на часто используемых столбцах:
- `objects.address`
- `objects.Company`
- `objects.dtype`
- `objects.status`
- `objects.manager`
- `objects.IMEI`
- Составной индекс: `objects(Company, dtype, status)`

## Тестовые данные

При первом запуске seeder добавляет:
- **Администратор**: login=`admin`, password=`admin123` (хешируется в MD5)
- **Тестовый объект**: Test Address 1 в Test City

## Миграции

Все миграции находятся в папке `database/migrations/`:

1. `2024_01_01_000000_create_users_table.php`
2. `2024_01_02_000000_create_objects_table.php`
3. `2024_01_03_000000_create_objects_install_data_table.php`
4. `2024_01_04_000000_create_log_table.php`
5. `2024_01_05_000000_create_csq_table.php`

### Запуск миграций

```bash
# Выполнить все миграции
php artisan migrate

# Откатить все миграции и выполнить заново
php artisan migrate:fresh

# Выполнить с добавлением тестовых данных
php artisan migrate:fresh --seed
```

## Дальнейшее развитие

Для полной функциональности системы требуется создать миграции и модели для динамических таблиц. На данный момент они создаются программно при добавлении нового объекта через соответствующую логику в контроллерах.
